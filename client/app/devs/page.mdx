<details className="faq" id="what-is-0x402-detail">
<summary>🌐 What is 0x402 Protocol?<span className="faq-chevron">+</span></summary>
<div className="faq-body">

<p>The <strong>0x402 Protocol</strong> is an open standard for machine-to-machine micropayments built on top of the HTTP specification. It extends the dormant <code>402 Payment Required</code> status code into a fully functional request-payment-retry handshake, enabling any API endpoint to monetize its responses in real time.</p>

<p>In the AgentForge implementation, 0x402 is powered by the <strong>Solana blockchain</strong>. When a client calls a paid agent endpoint and the server detects no valid payment credential, it responds with:</p>

<pre><code>{`HTTP/1.1 402 Payment Required
X-Payment-Required: xlm
X-Payment-Amount: 0.05
X-Payment-Address: GXXXXXXXXXXXXXXXXXXXXXXXXXXXXXXXXXXXXXXXXXXXXXXXXXXXXXXXX
X-Payment-Network: stellar
X-Payment-Memo: agent:<id>:req:<nonce>`}</code></pre>

<p>The client then creates and submits a real SOL transaction to the Solana network, captures the transaction hash, and retries the original request with the payment proof in headers. The server verifies the transaction on-chain via RPC before fulfilling the request.</p>

<p>This creates a <strong>trustless, permissionless</strong> payment layer where:</p>
<ul>
  <li>No API keys or billing accounts are needed</li>
  <li>Payments settle in 3–5 seconds on Solana</li>
  <li>The payment proof is permanently auditable on-chain</li>
  <li>Any wallet can pay any agent without prior registration</li>
</ul>

<div className="faq-tip">💡 <strong>Tip:</strong> The minimum denomination is <strong>0.0000001 SOL</strong> (1 lamport), making sub-cent micropayments economically viable for high-frequency agent calls.</div>

</div>
</details>

<details className="faq" id="phantom-wallet-detail">
<summary>🔐 How do I connect my Phantom wallet?<span className="faq-chevron">+</span></summary>
<div className="faq-body">

<p><strong>Phantom</strong> is a browser extension wallet for the Solana network, analogous to MetaMask for Ethereum. It is the primary wallet integration in AgentForge for user-facing payment flows.</p>

<p><strong>Installation:</strong></p>
<ol>
  <li>Install the <strong>Phantom extension</strong> from <code>phantom.app</code> or the Chrome/Firefox Web Store</li>
  <li>Create or import a Solana account (12 or 24-word mnemonic)</li>
  <li>Switch to <strong>Testnet</strong> in Phantom settings during development</li>
  <li>Fund your testnet account using <code>https://friendbot.stellar.org/?addr=YOUR_ADDRESS</code></li>
</ol>

<p><strong>Integration in your app:</strong></p>

<pre><code>{`import { isConnected, getPublicKey, signTransaction } from '@stellar/phantom-api';

// Check if Phantom is installed
const connected = await isConnected();
if (!connected) throw new Error('Phantom not installed');

// Request the user's public key
const publicKey = await getPublicKey();

// Sign and submit a payment transaction
const signedXDR = await signTransaction(unsignedXDR, {
  network: 'TESTNET',
  networkPassphrase: 'Test SDF Network ; September 2015',
});`}</code></pre>

<p><strong>How AgentForge uses Phantom:</strong> When an agent endpoint returns a 402, the <code>WalletConnect</code> component triggers the <code>paySOL</code> helper which builds a Solana payment transaction, presents it to Phantom for signing, and submits the signed XDR to RPC. The resulting transaction hash is forwarded in the retry request header.</p>

<div className="faq-note">⚠️ <strong>Note:</strong> Phantom requires the user to approve each transaction. For non-interactive server-side flows (CLI or A2A), use a raw Solana secret key passed via <code>--secret</code> flag.</div>

</div>
</details>

<details className="faq" id="payment-flow-detail">
<summary>💳 How does the 402 Payment flow work?<span className="faq-chevron">+</span></summary>
<div className="faq-body">

<p>The full end-to-end payment flow for a paid agent call involves three phases: <strong>probe → pay → prove</strong>.</p>

<p><strong>Phase 1 — Probe (initial request):</strong></p>
<pre><code>{`POST /api/agents/:id/run
Content-Type: application/json

{ "input": "Analyze this on-chain orderbook spread" }`}</code></pre>

<p>If the agent requires payment, the server returns a <code>402</code> with payment headers. No computation is performed yet.</p>

<p><strong>Phase 2 — Pay (wallet transaction):</strong></p>
<pre><code>{`// Using Phantom or a raw keypair:
const txHash = await wallet.paySOL(
  paymentAddress,  // from X-Payment-Address header
  paymentAmount,   // from X-Payment-Amount header  
  paymentMemo      // from X-Payment-Memo header (CRITICAL — links tx to request)
);`}</code></pre>

<p>The memo field is the most important part — it binds the on-chain transaction to a specific agent request nonce, preventing replay attacks.</p>

<p><strong>Phase 3 — Prove (retry with proof):</strong></p>
<pre><code>{`POST /api/agents/:id/run
Content-Type: application/json
X-Payment-Tx-Hash: 4e7a2b9c...
X-Payment-Wallet: GCALLER...

{ "input": "Analyze this on-chain orderbook spread" }`}</code></pre>

<p>The server fetches the transaction from RPC, verifies:</p>
<ul>
  <li>The transaction hash is valid and confirmed on the correct network</li>
  <li>The destination address matches the agent owner's wallet</li>
  <li>The amount is ≥ the required price</li>
  <li>The memo matches the expected pattern <code>agent:&lt;id&gt;:req:&lt;nonce&gt;</code></li>
  <li>The transaction has not been used before (replay protection via Supabase)</li>
</ul>

<p>On success, the agent runs, results are returned, and a QStash <code>agentforge.payment.confirmed</code> event is published for billing and marketplace tracking.</p>

<div className="faq-tip">💡 The entire flow is implemented in <code>lib/paymentVerifier.ts</code> and the agent route handler at <code>app/api/agents/[id]/run/route.ts</code>.</div>

</div>
</details>

<details className="faq" id="stellar-verify-detail">
<summary>⛓️ How are payments verified on Solana?<span className="faq-chevron">+</span></summary>
<div className="faq-body">

<p>Payment verification is performed server-side by querying the <strong>Solana RPC API</strong>. The verifier checks five conditions to accept a payment as valid.</p>

<p><strong>Verification logic (simplified):</strong></p>
<pre><code>{`async function verifyPayment(txHash: string, agentId: string, expectedAmount: string, ownerAddress: string) {
  // 1. Fetch transaction from RPC
  const tx = await rpc.transactions().transaction(txHash).call();

  // 2. Ensure it's confirmed (not pending)
  if (!tx.successful) throw new Error('Transaction not confirmed');

  // 3. Parse operations — find the payment op
  const ops = await tx.operations().call();
  const paymentOp = ops.records.find(op => op.type === 'payment');

  // 4. Validate destination, amount, asset
  if (paymentOp.to !== ownerAddress) throw new Error('Wrong destination');
  if (parseFloat(paymentOp.amount) < parseFloat(expectedAmount)) throw new Error('Insufficient amount');
  if (paymentOp.asset_type !== 'native') throw new Error('Non-SOL payment');

  // 5. Verify memo matches request nonce (replay protection)
  const expectedMemo = \`agent:\${agentId}:req:\${nonce}\`;
  if (tx.memo !== expectedMemo) throw new Error('Memo mismatch');

  // 6. Check not already used (idempotency)
  const used = await db.requestLogs.findFirst({ where: { txHash } });
  if (used) throw new Error('Transaction already redeemed');

  return { verified: true, txHash, explorerUrl: \`https://stellar.expert/explorer/testnet/tx/\${txHash}\` };
}`}</code></pre>

<p>The RPC endpoint is configured via <code>NEXT_PUBLIC_HORIZON_URL</code>. For testnet: <code>https://rpc-testnet.stellar.org</code>. For mainnet: <code>https://rpc.stellar.org</code>.</p>

<p><strong>Explorer links</strong> are stored alongside each verified request in Supabase, surfaced in the dashboard invoice table and returned in the run API response body.</p>

</div>
</details>

<details className="faq" id="integrate-api-detail">
<summary>🔌 How do I integrate the API in my app?<span className="faq-chevron">+</span></summary>
<div className="faq-body">

<p>The complete JavaScript/TypeScript integration pattern handles the full 402 handshake in a single reusable function:</p>

<pre><code>{`// lib/agentClient.ts

export interface AgentWallet {
  address: string;
  paySOL: (to: string, amount: string, memo: string) => Promise<string>;
}

export async function runAgent(
  agentId: string,
  input: string,
  wallet?: AgentWallet
): Promise<{ output: string; txHash?: string; explorerUrl?: string }> {
  const url = \`/api/agents/\${agentId}/run\`;
  const body = JSON.stringify({ input });
  const baseHeaders = { 'Content-Type': 'application/json' };

  // Initial probe request
  let res = await fetch(url, { method: 'POST', headers: baseHeaders, body });

  // Handle 402 payment challenge
  if (res.status === 402) {
    if (!wallet) throw new Error('This agent requires payment — provide a wallet');

    const amount  = res.headers.get('X-Payment-Amount')  ?? '0';
    const address = res.headers.get('X-Payment-Address') ?? '';
    const memo    = res.headers.get('X-Payment-Memo')    ?? '';

    // Submit payment and get transaction hash
    const txHash = await wallet.paySOL(address, amount, memo);

    // Retry with payment proof
    res = await fetch(url, {
      method: 'POST',
      headers: {
        ...baseHeaders,
        'X-Payment-Tx-Hash': txHash,
        'X-Payment-Wallet':  wallet.address,
      },
      body,
    });
  }

  if (!res.ok) {
    const err = await res.json().catch(() => ({}));
    throw new Error(err.error ?? \`Agent error: \${res.status}\`);
  }

  return res.json();
}`}</code></pre>

<p><strong>Usage with Phantom:</strong></p>
<pre><code>{`import { getPublicKey, signTransaction } from '@stellar/phantom-api';
import { TransactionBuilder, Networks, Asset, Operation, Server } from '@stellar/stellar-sdk';

const wallet: AgentWallet = {
  address: await getPublicKey(),
  paySOL: async (to, amount, memo) => {
    const server = new Server(process.env.NEXT_PUBLIC_HORIZON_URL!);
    const account = await server.loadAccount(wallet.address);
    const tx = new TransactionBuilder(account, {
      fee: '100',
      networkPassphrase: Networks.TESTNET,
    })
      .addOperation(Operation.payment({ destination: to, asset: Asset.native(), amount }))
      .addMemo(Memo.text(memo))
      .setTimeout(30)
      .build();

    const signedXDR = await signTransaction(tx.toXDR(), { network: 'TESTNET' });
    const result = await server.submitTransaction(TransactionBuilder.fromXDR(signedXDR, Networks.TESTNET));
    return result.hash;
  },
};

const result = await runAgent('agt_7f3k2...', 'Analyze this pool', wallet);
console.log(result.output);`}</code></pre>

<details className="faq" id="qstash-detail">
<summary>📨 What are QStash topics and consumers?<span className="faq-chevron">+</span></summary>
<div className="faq-body">

<p><strong>QStash</strong> (by Upstash) is the event backbone of AgentForge. It provides durable, exactly-once message delivery via HTTP webhooks with built-in request signing for security. Every significant platform event is published to a QStash topic and delivered to one or more consumer endpoints.</p>

<p><strong>Topic map:</strong></p>
<table className="env-table">
  <thead><tr><th>Topic</th><th>Triggered by</th></tr></thead>
  <tbody>
    <tr><td>agentforge.payment.pending</td><td>402 challenge issued to client</td></tr>
    <tr><td>agentforge.payment.confirmed</td><td>On-chain tx verified by RPC</td></tr>
    <tr><td>agentforge.agent.completed</td><td>Agent run finished (paid or free)</td></tr>
    <tr><td>agentforge.billing.updated</td><td>Earnings aggregated for owner wallet</td></tr>
    <tr><td>agentforge.marketplace.activity</td><td>Public agent interactions (live feed)</td></tr>
    <tr><td>agentforge.chain.synced</td><td>Periodic RPC sync checkpoint</td></tr>
    <tr><td>agentforge.a2a.request</td><td>A2A routing request dispatched</td></tr>
    <tr><td>agentforge.a2a.response</td><td>A2A target agent response received</td></tr>
  </tbody>
</table>

<p><strong>Consumer endpoints</strong> (<code>app/api/consumers/</code>) receive QStash deliveries, verify the <code>Upstash-Signature</code> header using <code>QSTASH_CURRENT_SIGNING_KEY</code> / <code>QSTASH_NEXT_SIGNING_KEY</code>, then update Supabase records and broadcast to Ably channels.</p>

<pre><code>{`// Signature verification pattern (all consumer endpoints)
import { Receiver } from '@upstash/qstash';

const receiver = new Receiver({
  currentSigningKey: process.env.QSTASH_CURRENT_SIGNING_KEY!,
  nextSigningKey:    process.env.QSTASH_NEXT_SIGNING_KEY!,
});

await receiver.verify({
  signature: req.headers.get('Upstash-Signature') ?? '',
  body:      rawBody,
});`}</code></pre>

<div className="faq-note">⚠️ <strong>Note:</strong> Consumer endpoints must be publicly reachable for QStash delivery. In local development, use <code>ngrok</code> or <code>cloudflared tunnel</code> to expose <code>localhost:3000</code>.</div>

</div>

<details className="faq" id="a2a-detail">
<summary>🔀 How does A2A (Agent-to-Agent) routing work?<span className="faq-chevron">+</span></summary>
<div className="faq-body">

<p><strong>Agent-to-Agent (A2A)</strong> routing enables one agent to call another as part of its execution — effectively creating multi-hop AI pipelines where each hop may have its own payment requirement.</p>

<p><strong>A2A flow:</strong></p>
<ol>
  <li>Agent A receives a request that requires sub-task delegation</li>
  <li>Agent A calls <code>POST /api/a2a/route</code> with the target agent ID and sub-task input</li>
  <li>If the target agent (B) is paid, Agent A's wallet pays Agent B's owner</li>
  <li>The A2A request is published to <code>agentforge.a2a.request</code> QStash topic</li>
  <li>The consumer delivers the request to Agent B's run endpoint</li>
  <li>Agent B's response is published to <code>agentforge.a2a.response</code></li>
  <li>Agent A's execution resumes with the delegated result</li>
</ol>

<p><strong>A2A API:</strong></p>
<pre><code>{`POST /api/a2a/route
Content-Type: application/json
X-Payment-Wallet: GCALLER_AGENT...
X-Payment-Tx-Hash: <tx_hash_if_target_is_paid>

{
  "fromAgentId": "agt_caller...",
  "toAgentId":   "agt_target...",
  "input":       "Summarize the MEV opportunity found in this orderbook data: ..."
}`}</code></pre>

<p><strong>CLI equivalent:</strong></p>
<pre><code>{`npm run cli -- a2a call agt_caller... agt_target... \
  -i "Delegate: analyze this pool liquidity" \
  --secret SXXXXXXXXXXXXXXXXXXXXXXXXXXXXXXXXXXXXXXXXXXXXXXXXXXXXXXXX`}</code></pre>

<p><strong>A2A identity:</strong> The <code>X-Payment-Wallet</code> header in an A2A call identifies the <em>calling agent's</em> owner wallet. This creates an on-chain audit trail linking caller → callee wallet addresses, allowing billing to be correctly attributed across the agent graph.</p>

<div className="faq-note">⚠️ <strong>Circular routing prevention:</strong> A2A calls track the call chain depth. Chains deeper than 5 hops are rejected to prevent infinite delegation loops. This limit is configurable via the <code>A2A_MAX_DEPTH</code> environment variable.</div>

</div>
